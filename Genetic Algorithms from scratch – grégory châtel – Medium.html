Genetic Algorithms from scratch

While I was studying Computer Science at university, I came across genetic algorithms, a way to find very good approximations to hard problems. This family of algorithm is inspired by biological evolution in which a population evolves from one generation to the next one by selecting the fittest individuals, reproducing and mutating.

I was and still am amazed by how such a simple idea could help to solve (and by solving I mean finding a good approximation) problems this complicated. Here are a few examples of problems that can be tackled with this method:

  • The Travelling Salesman Problem: You have to find an order to visit a set of cities that have the lowest gas consumed possible.
  • Scheduling problem: You are playing a real time strategy video game and you want to have a specific set of units as soon as possible. The goal is to find the most efficient build order to achieve this task. Some people applied this concept to Starcraft 2.
  • Evolving a population of creature to complete a specific task like walking or jumping.

In this blog post, we will build a very simple genetic algorithm to find a reasonable solution to the Traveling Salesman Problem (TSP).

First, let’s state the TSP properly. The input of this problem is a set of cities which looks like this:

From that, we want to find an itinerary that goes through these cities using as little gas as possible. A good solution for this problem could look like this:

To be precise, we try to find an itinerary that minimizes the sum of Euclidean distances between consecutive cities.

It is important to note that the solution presented above is not THE best one for this problem. The TSP is what computer scientists call a NP-hard problem which means that there probably does not exist a polynomial algorithm to solve it. Genetics Algorithms do not provide exact solutions but heuristics (approximate solution). In a sense, we are trading accuracy for speed, which is great since there are a lot of super hard problems where a good solution is enough to work with.

Now that the problem is properly stated, let’s do the parallel between biological concepts and elements of the Traveling Salesman Problem.

When trying to apply genetic algorithms, the first thing to do is to build a mapping between a list of biological concepts and details of our problem. The evolutionary elements are the following ones:

  • Individual: an individual of our specie.
  • Population: a group of individual which will evolve over time.
  • Fitness measure: a way to quantify how adapted an individual is to its environment.
  • Reproduction: an operation that creates a new individual from two parents in a way that he takes characteristics from both his parents.
  • Selection: a way to choose which individual will reproduce.
  • Mutation: a operation that slightly modifies an individual in order to keep diversity in the genome of the population and find creative ways to increase fitness that other individuals do not possess.

Now, let’s build this mapping!

In our case, an individual will be an itinerary (the order of visit) that goes through each of the cities. The population will be a set of individuals, in our case, itineraries. The fitness measure of an individual is the total distance traveled in this path. This metric should represent what we want to optimize for our problem.

Now the more interesting concepts:

  • We define the reproduction of two path as follows: we take the order of visit of half the cities from the first path and the order of visit of the other half from the second path. Let’s see it on an example.

First path = [A, F, G, C, D, H, B, E]

Second path = [E, D, C, B, F, H, G, A]

We select 4 cities from the first path: [F, G, H, E]

and the remaining 4 from the second one: [D, C, B, A]

new individual = [F, G, H, E, D, C, B, A]

There are many different ways to define the reproduction of two paths, this is the one I chose.

  • To select individuals for reproduction, we chose the first one from the top 15% fittest individuals and the second one randomly. Once again, there are many different ways to select individuals, I chose this one because it is really simple and powerful enough.
  • In our case, the mutation consists in taking a random slice of the itinerary and reversing it. Let’s see it on an example:

Individual = [F, G, H, E, D, C, B, A]

We randomly choose a slice: [D, C, B]

And reverse it: [B, C, D]

The mutated individual is then: [F, G, H, E, B, C, D, A]

As always, there are many possible choices for the mutation. I thought of this one while trying to solve a pattern that reproduction did not seem able to deal with (crosses between paths).

Now that we have our mapping, we can implement the core of our genetic algorithm.

The pseudo-code of the algorithm is the following:

population = randomly create initial population()
for each generation:
    survivors number = population size - number of reproduction -
                       number of mutation
    new population = top 'survivors number' fittest of population
    for 'number of reproduction' times:
        first individual = select individual from top 15% fittest
        second individual = randomly select an individual 
        new individual = reproduce(first individual, second
                                   invidual)
        new population = new population + new individual
    for 'number of mutation' times:
        individual to mutate = randomly select an individual
        new individual = mutate(individual to mutate)
        new population = new population + new individual

And that’s all!
